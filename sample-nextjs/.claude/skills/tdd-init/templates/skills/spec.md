# /spec — 仕様を書く＋精査する（{{PROJECT_NAME}}）

> 仕様駆動 TDD の入口。あなたはテスト設計を考えなくてよい。**ラフな要望を渡せば、TDD に投入できる仕様書**を起こす。
> このアプリの責務: {{DOMAIN_SUMMARY}}

## 入力
`$ARGUMENTS` = 作りたい機能のラフな説明 / チケットの本文・パス・URL。空なら「何を作りたいか」を一言もらう。

## 着手前に読む
- `.claude/tdd/rules/spec-conventions.md` … このプロジェクトの仕様書フォーマット
- `.claude/tdd/rules/test-conventions.md` … どんなテストに落ちるか（仕様の粒度の参考）
- 関連する既存コード/既存仕様（同種機能があれば構造を踏襲）

## 手順

### 1. 取り込み
ラフな要望・チケットを読む。関連する既存機能・データ・画面をプロジェクトから把握する。

### 2. 仕様ドラフトを起こす
`spec-conventions.md` の型に沿って構造化する。最低限:
- **目的 / 背景** … なぜ作るか
- **スコープ / 非対象** … やること・やらないこと
- **ユースケース / ユーザーストーリー** … 誰が何をして何を得るか
- **受け入れ条件**（Given–When–Then で列挙。これがそのままテストになる）
- **正常系 / 境界 / 異常系** … 各条件の具体ケース
- **E2Eシナリオ候補** … ユーザー操作フロー（あれば。Playwright 対象）
- **影響範囲** … 触る画面・API・データ（{{SRC_LAYOUT}} を踏まえて）

### 3. 精査する（ここが価値）
ドラフトに対し、**実装に入る前に潰すべき点**を自分で洗い出す:
- 曖昧な語（「うまく」「適切に」等）を具体値・具体条件に落とせているか
- 抜けている異常系・境界（空・上限・権限・並行・失敗時の挙動）
- 矛盾・前提の食い違い
- 受け入れ条件が**検証可能（テストにできる）**な粒度か

判断が要る不明点だけ**まとめて1回**ユーザーに確認する（小出しにしない）。自明な補完は確認せず埋める。

### 4. 出力
`.claude/tdd/specs/<feature-slug>.md` に書き出す（slug は機能名のケバブケース）。

### 5. 報告
仕様書のパスと、残課題があれば明示し、次の一歩を伝える:
> 仕様を `.claude/tdd/specs/<slug>.md` に起こしました。`/tdd .claude/tdd/specs/<slug>.md` で実装に入れます。

## ルール
- **テストにできない曖昧仕様のまま終えない。** 受け入れ条件は Given–When–Then で検証可能に。
- 実装はしない（ここは仕様フェーズ）。コードは書かない。
- 不明点の確認は1回にまとめる。楽に進められることを優先しつつ、的外れ防止の精査だけは省かない。
