# 仕様書の書き方 — {{PROJECT_NAME}}

> `/spec` が起こす仕様書のフォーマット。受け入れ条件は**そのままテストになる粒度**で書く。
> アプリの責務: {{DOMAIN_SUMMARY}}

## 置き場
`.claude/tdd/specs/<feature-slug>.md`（slug は機能名のケバブケース）。

## テンプレート

```markdown
# <機能名>

## 目的 / 背景
なぜ作るか。解決したい課題。

## スコープ / 非対象
- やること
- やらないこと（明示的に除外）

## ユースケース
誰が・何をして・何を得るか（1〜複数）。

## 受け入れ条件（Given–When–Then）
- [ ] Given <前提> When <操作> Then <期待結果>
- [ ] ...
（各条件はそのまま1つ以上のテストになる。検証可能な具体値で書く）

## 正常系 / 境界 / 異常系
- 正常: ...
- 境界: 空 / 上限 / ゼロ / 最大長 など
- 異常: 不正入力 / 権限なし / 失敗時の挙動 / 並行

## E2E シナリオ候補（あれば）
ユーザー操作フロー（Playwright 対象）。

## 影響範囲
触る画面 / API / データ / モジュール（{{SRC_LAYOUT}} を踏まえて）。

## 残課題 / 未確定
精査で残った確認事項。
```

## 守ること

- **曖昧語を残さない**: 「うまく」「適切に」「いい感じに」は具体条件・具体値に落とす。
- **受け入れ条件は検証可能に**: Given–When–Then で、テストに変換できる粒度。
- **異常系・境界を必ず1つ以上**書く（正常系だけの仕様にしない）。
- 未確定点は「残課題」に明示し、放置せず `/spec` の精査で潰す。
