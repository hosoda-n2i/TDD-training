# テスト規約 — sample-nextjs

> `/tdd-init` が検出した、このプロジェクトのテストの書き方。
> **既存テストが正**。新しく書くテストは既存に合わせる。迷ったら既存のテストファイルを 1 つ開いて真似る。

## 配置と命名

- **配置**: ソースファイルと同階層（例: `src/lib/math.ts` → `src/lib/math.test.ts`）
- **ファイル命名**: `*.test.ts` / `*.test.tsx`（コンポーネントは `.test.tsx`）
- 1 つのテスト対象（関数 / モジュール / クラス）に対応するテストファイルを 1 つ作るのが基本。

## テストの構造（AAA）

各テストは **Arrange → Act → Assert** の 3 ブロックで書く。

- **Arrange**: 入力・前提・モックを準備する。
- **Act**: テスト対象を 1 回だけ呼ぶ。
- **Assert**: 結果を検証する。1 テスト = 1 つの論理的な振る舞いの検証。

テスト名は「何を検証しているか」が読み取れる文にする（実装でなく **振る舞い** を記述する）。

## アサーション

- スタイル: `expect(actual).toBe(expected)` / `expect(actual).toEqual(expected)`
- 等価・例外・非同期など、検証したい性質に合ったマッチャ/アサーションを使う。
- 1 つのテストでアサーションを盛り過ぎない。複数の性質を見たいならテストを分ける。

## モック / スタブ / フェイク

- 手法: `vi.mock()` / `vi.fn()` / `vi.spyOn()`（Vitest 組み込み）
- **モックは外部境界（I/O・ネットワーク・時刻・乱数・DB）に限定する。** 自分のドメインロジックはモックしない。
- 過剰なモックは「実装をなぞるだけのテスト」を生む。可能なら本物を使い、必要な箇所だけ差し替える。

## テストデータ / フィクスチャ

- 繰り返す準備はファクトリ/ヘルパに切り出す（既存のものがあれば再利用）。
- マジックナンバー・固定文字列には意味のある名前を付ける。

## 何をテストするか

- **する**: 公開された振る舞い・境界値・異常系・回帰（過去のバグ）。
- **しない**: 言語/ライブラリ自体の挙動、private 実装の詳細、自明な getter/setter。
- カバレッジは目安であって目的ではない。**意味のある振る舞いを網羅する** ことを優先する。

## このプロジェクト固有のメモ

- テスト環境: `jsdom`（`vitest.config.ts` で設定済み）
- セットアップファイル: `vitest.setup.ts`（`@testing-library/jest-dom` のマッチャが使える）
- パスエイリアス: `@/` → `src/`（例: `import { sum } from '@/lib/math'`）
- React コンポーネントのテストは `@testing-library/react` を使う
