# TDD ワークフロー — sample-nextjs

> このファイルは `/tdd-init` が生成した。プロジェクト固有の値は検出結果で埋めてある。
> 手で更新してもよいが、再生成すると上書きされる（`progress.md` は除く）。

主要言語: **TypeScript** ／ テストランナー: **Vitest**

## 基本原則

1. **テストを先に書く。** プロダクションコードは「失敗しているテストを通すため」にだけ書く。
2. **一度に 1 つの振る舞い。** 大きな機能は小さな振る舞いに割って、1 サイクル 1 振る舞い。
3. **3 つの状態を行き来する**:
   - 🔴 **Red** … 次の振る舞いを表す失敗するテストを 1 本書く。期待した理由で失敗することを確認する。
   - 🟢 **Green** … そのテストを通す **最小限** のコードを書く。きれいさは後回し、まず通す。
   - 🔵 **Refactor** … テストが緑のまま、重複除去・命名改善・構造整理。**振る舞いは変えない**。
4. **緑でない状態を放置しない。** コミットは原則「緑かつ refactor 済み」の時点で行う。

## 1 サイクルの具体手順

### 🔴 Red
1. 実装する振る舞いを 1 文で言語化する（「〜のとき〜を返す/する」）。
2. その振る舞いを表すテストを **1 本** 書く（配置・命名は `conventions.md` に従う）。
3. テストを実行して **失敗する** ことを確認する:
   ```
   npx vitest run <path/to/file.test.ts>
   ```
4. 失敗が「未実装だから」であって、テストの書き間違い（typo・import ミス）でないことを確認する。

### 🟢 Green
1. テストを通す **最小限** の実装を書く。ベタ書き・固定値返しでもこの段階では可。
2. テストを実行して通すことを確認する:
   ```
   npx vitest run <path/to/file.test.ts>
   ```
3. 既存テストを壊していないか全実行で確認する:
   ```
   npm test
   ```

### 🔵 Refactor
1. テストが緑のまま、コードとテストの両方を整理する（重複・命名・責務）。
2. 変更のたびに再実行して緑を維持する。
3. `conventions.md` の規約から外れていないか確認する。

### ✅ サイクル終了
1. lint / 型チェックがあれば通す（`commands.md` 参照）。
2. `progress.md` に 1 行記録する（実装した振る舞い・テスト名）。
3. その単位でコミットする（プロジェクトのコミット規約に従う。無ければ `test:` / `feat:` 等の慣習に従う）。

## 進め方のガイド

- **テストが大きすぎたら割る。** 1 本のテストで複数の振る舞いを検証していたら、Red を分割する。
- **テストが通らず詰まったら戻す。** 直前の緑に戻り、振る舞いをさらに小さく割り直す。
- **三角測量**: 仮実装（固定値）→ 2 つ目のテスト追加 → 一般化、で正しい実装に寄せていく。
- **テストの可読性もプロダクションコードと同等に保つ**。テストは仕様書でもある。

詳細は `conventions.md`（書き方）と `commands.md`（実行）を参照。
