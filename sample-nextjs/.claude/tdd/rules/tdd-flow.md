# 仕様駆動 TDD の規律 — sample-nextjs

> `/tdd-init` が生成した常時適用ルール。このプロジェクトの開発はこの規律に従う。
> アプリの責務: TDD トレーニング用 Next.js サンプルアプリ。App Router ページと src/lib の純粋ロジックを組み合わせて機能を実装する。

## 基本姿勢

- **仕様 → テスト → 実装** の順。実装から書き始めない。
- 新機能・修正は **`/spec`（仕様を起こし精査）→ `/tdd`（実装）** の2段で進める。
- テストは仕様の受け入れ条件から導く。**「振る舞いを思いつく」のではなく「仕様から抽出する」**。
- テストを先に書き、**期待どおり赤**を確認してから実装する。
- **緑でない状態でコミットしない。** 実装はテストを通すためだけに書き、テストを実装に合わせて歪めない。

## テストレベル（詳細は `test-strategy.md`）

- **unit（Vitest）**: 純粋ロジック・関数・コンポーネント単体。最も多く、速い。
- **integration**: 複数モジュール・route handler・DB 等をまたぐ。
- **E2E（Playwright）**: ユーザー操作フロー・画面横断の受け入れ条件。

仕様の各受け入れ条件は、検証に最も適したレベルに割り当てる。E2E で全部やろうとしない（遅く脆い）。

## フロー（`/tdd` が駆動する）

1. 仕様からテスト洗い出し（E2E シナリオを最初に地図化）
2. 実装計画立案（テストケース ↔ 実装箇所の対応）
3. テスト先行 → **テスト自体をレビュー**
4. 仕様内容の実装（最小実装 → リファクタ）
5. 既存テストの回帰確認
6. E2E / 各種テストの通過確認
7. 一通りの動作確認
8. 実行時不具合のレビュー → 修正 → 再実行

## 守ること

- コマンドは `.claude/tdd/commands.md` のものだけを使う。
- テストの配置・命名・モック方針は `.claude/tdd/rules/test-conventions.md` に従う。
- 確認ステップは最小限（実装計画の提示・E2E導入の可否）。それ以外は自己完結で進める。
- `push` はユーザーが明示したときだけ。コミットは受け入れ条件（論理単位）ごと。
