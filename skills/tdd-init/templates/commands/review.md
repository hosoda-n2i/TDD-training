## Context

!`git diff --stat HEAD 2>&1`

## Your task

差分を以下の観点で点検する。diff に明らかに無関係なカテゴリはスキップしてよい。

### Step 1: 差分を読む
!`git diff HEAD 2>&1`

### Step 2: 観点別チェックリスト

#### 1. 仕様充足
- 受け入れ条件（仕様 / E2E spec / integration test に書かれているもの）を**全て**満たしているか
- 受け入れ条件にあるのに**テストが書かれていない**ものはないか
- 受け入れ条件に無いのに**実装が増えている**箇所はないか（スコープ外）

#### 2. 可読性・保守性
- 変数名 / 関数名が振る舞いを表しているか
- 1 関数 1 責務になっているか
- マジックナンバー / マジックストリングが定数化されているか
- ネストが深すぎないか（早期 return / ガード句で改善できないか）
- dead code が残っていないか
- 型定義が適切か（`any` の濫用が無いか、export には明示的な型が付いているか）

#### 3. ロジック・正確性
- エッジケースが考慮されているか（空配列・null/undefined・0・空文字・最大長）
- 境界値の処理が正しいか
- 条件分岐に漏れがないか（else / default）
- ループの終了条件が正しいか
- 非同期処理の `await` 漏れ / Promise 処理
- 日付・タイムゾーンの扱い

#### 4. セキュリティ
- XSS 対策（ユーザー入力のサニタイズ / `dangerouslySetInnerHTML` の濫用）
- SQL/NoSQL インジェクション（パラメータ化クエリ / ORM のクエリビルダ）
- 認証・認可（**IDOR**: 他人のリソースに触れない）
- 入力バリデーション（**サーバ側でも** Zod 等で検証）
- 機密情報がログ/レスポンスに漏れていないか
- 依存パッケージの既知脆弱性

#### 5. パフォーマンス
- N+1 クエリ
- 不要な再レンダリング（`useMemo`/`useCallback`/`memo` の適切な使用）
- 大量データの処理にページネーション / 無限スクロール
- 画像 / アセットの最適化
- バンドルサイズ（不要 import / tree shaking）
- DB インデックス
- メモリリーク（イベントリスナーの解除漏れ / タイマーの clear 漏れ）

#### 6. エラーハンドリング
- 例外がユーザーに分かりやすい形で返るか
- 認証失敗 → リダイレクト、データ未検出 → 404 等の分岐が適切か
- ネットワークエラー / タイムアウトのハンドリング
- リアルタイム購読（onSnapshot 等）のエラーコールバック

#### 7. テスト
- 追加 / 変更したコードに対するテストが書かれているか
- 異常系のテストケースがあるか
- テスト間の独立性（共有可変状態が無いか）
- モックの使い方が適切か（**自分のドメインロジックはモックしない**）
- カバレッジが大きく落ちていないか
- Arrange–Act–Assert 構造になっているか

#### 8. API / Server Action 設計
- Server Action と API Route の使い分けが規約通りか
- 認可ミドルウェアが適切に挟まっているか
- Zod スキーマで入力検証されているか
- キャッシュ無効化（`revalidatePath` 等）が適切か
- レスポンス形式の統一

#### 9. データベース
- マイグレーションが正しい（ロールバック可能か）
- テーブル設計の正規化 / 意図的非正規化
- 制約（FK / NOT NULL / UNIQUE）の設定
- 型の選定（boolean を integer で代用していないか 等）
- トランザクションが必要な処理で使われているか

#### 10. フロントエンド固有
- Server Component / Client Component の境界（`'use client'` を葉に押し込めているか）
- レスポンシブ
- 状態管理（Server Component 優先、Client 状態は最小限）
- フォーム検証（React Hook Form + Zod 等）
- UI テキスト・メッセージの言語（プロジェクト規約に合っているか）
- アクセシビリティ（セマンティック HTML / `aria-label`）

### Step 3: 報告

各指摘について以下を出力:

| 観点 | 重要度 | ファイル:行 | 何が問題か | 修正案 |

重要度の凡例:
- **error** — 必ず直す（バグ / セキュリティ / 仕様未充足）
- **warning** — 直すべき（可読性 / パフォーマンス / 規約違反）
- **info** — 検討推奨（リファクタ余地）

最後にサマリ: 合計件数、重要度別内訳。

## Rules

- diff 外のファイルを巻き込んだ「ついでに直す」提案はしない。スコープを守る。
- **指摘の根拠は具体的に**（「読みづらい」ではなく「ネストが 4 段で早期 return できる」）。
- 推測で「公式が推奨」「ベストプラクティス」と書かない。プロジェクト内の既存パターン or 一次情報（doc URL）を引く。
