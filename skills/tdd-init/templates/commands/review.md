## Context

!`git diff --stat HEAD 2>&1`

## Your task

差分を以下の観点で点検する。diff に明らかに無関係なカテゴリはスキップしてよい。

### Step 1: 差分を読む
!`git diff HEAD 2>&1`

### Step 2: 観点別チェックリスト

1. **仕様充足** — 受け入れ条件に対応するテストが全てあるか。テスト漏れ・スコープ外実装が無いか。
2. **可読性・保守性** — 命名・1関数1責務・マジック定数化・ネスト深さ・dead code・`any` 濫用。
3. **ロジック・正確性** — エッジケース（null/0/空/最大長）・境界値・条件漏れ・`await` 漏れ・日付タイムゾーン。
4. **セキュリティ** — XSS・SQLi・IDOR・サーバ側バリデーション・機密漏洩・脆弱依存。
5. **パフォーマンス** — N+1・不要再レンダリング・ページネーション・バンドルサイズ・DB インデックス・メモリリーク。
6. **エラーハンドリング** — エラーがユーザーに届くか・404/リダイレクト分岐・タイムアウト・購読エラー。
7. **テスト** — 変更コードへのテスト・異常系・テスト間独立・モック適切・カバレッジ・AAA 構造。
8. **API / Server Action 設計** — Server Action / API Route 使い分け・認可・Zod・キャッシュ無効化・レスポンス統一。
9. **データベース** — マイグレーション正当性・制約・正規化・型選定・トランザクション。
10. **フロントエンド固有** — SC/CC 境界・レスポンシブ・状態最小化・フォーム検証・UI 言語・アクセシビリティ。

### Step 3: 報告

各指摘について以下を出力:

| 観点 | 重要度 | ファイル:行 | 何が問題か | 修正案 |

重要度の凡例:
- **error** — 必ず直す（バグ / セキュリティ / 仕様未充足）
- **warning** — 直すべき（可読性 / パフォーマンス / 規約違反）
- **info** — 検討推奨（リファクタ余地）

最後にサマリ: 合計件数、重要度別内訳。

## Rules

- diff 外のファイルを巻き込んだ「ついでに直す」提案はしない。スコープを守る。
- **指摘の根拠は具体的に**（「読みづらい」ではなく「ネストが 4 段で早期 return できる」）。
- 推測で「公式が推奨」「ベストプラクティス」と書かない。プロジェクト内の既存パターン or 一次情報（doc URL）を引く。
