# テスト戦略（dual-loop TDD） — {{PROJECT_NAME}}

> `/e2e` と `/tdd` が「どの受け入れ条件をどのレベルで test するか」を判断するための指針。
> 原則: **下位レベルで test できるものは下位で**。E2E に寄せ過ぎない（遅く脆い）。
> **逆に全部 unit に倒すのも禁止**。UI を伴うなら E2E、層またぎなら integration を必ず書く。

## Dual-loop の構造

```
外側 (E2E)        …… ユーザー操作フロー全体 = acceptance criteria
内側 (unit / integration) …… 受け入れ条件ごとに 1 サイクル
```

- **外側ループ**は `/e2e <機能名>` で先に書く（RED）。これが「機能完成」の物理的な定義になる。
- **内側ループ**は `/tdd <e2e spec / 機能説明>` で受け入れ条件を 1 つずつ緑にしていく。
- UI を伴わない機能（純ロジック / バックエンド API のみ等）は、**integration test が外側ループの役割を果たす**。`/tdd` から直接入る。

## レベルの選び方

| レベル | 対象 | 例 | ツール |
|--------|------|----|--------|
| **unit** | 純粋ロジック・関数・1 コンポーネント | 金額計算 / バリデーション / 整形 / 単一コンポーネントの表示・操作 | Vitest (+ RTL) |
| **integration** | 複数モジュール・route handler・DB / 外部 I/O 境界をまたぐ | Server Action が入力を検証し DB に保存して応答する | Vitest（実 DB / 認証モック） |
| **E2E** | 画面をまたぐユーザー操作フロー | ログイン → 一覧 → 詳細で目的を達成 | Playwright |

## 割り当ての判断

| この条件を満たすなら… | このレベル |
|----------------------|-----------|
| 「ある入力 → ある出力」で閉じる | **unit** |
| 「複数の部品が連携して」正しく動く（API ↔ DB / Service ↔ Repo / 認可境界等） | **integration** |
| 「ユーザーが画面を操作して」目的を達成できる | **E2E**（1 機能あたり主要フロー 1〜数本に絞る） |

- 同じ条件を複数レベルで重複検証しない。**一番安く検証できるレベル**を選ぶ。
- E2E は遅く脆いので **happy path 中心**。境界・異常系は unit / integration に寄せる。
- ただし**認証 / 認可の異常系**は E2E でも 1 本確認する価値がある（unit のモックでは検出できない実環境のずれ）。

## このプロジェクトでの適用状態（`/tdd-init` が実際に用意したもの）

> **「未実装・必要になったら導入」とは書かない。** 基盤を用意したなら「用意済み」、本当に対象外なら「対象外」と理由付きで明記。

- **unit**: 常時利用（Vitest 導入済み）
- **integration（境界モック）**: Vitest 上で外部境界（{{INTEGRATION_BOUNDARIES}}）をモック / 差し替えして検証。追加基盤は不要。
- **integration（実 DB）**: {{INTEGRATION_DB_STATUS}}  <!-- 例: 「テスト DB(docker-compose.test.yml)＋globalSetup 用意済み」/「対象外（実 DB アクセスなし）」 -->
- **E2E**: {{E2E_STATUS}}  <!-- 例: 「Playwright＋storageState ログイン基盤 用意済み、e2e/ に配置」/「対象外（画面なし）」 -->

「対象外」と書くのは、**本当に構造的に存在しない場合**（画面が無い／DB が無い 等）だけ。**面倒・後回しを理由に「対象外」にしない。**
