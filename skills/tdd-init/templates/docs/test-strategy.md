# テスト戦略（どの要件をどのレベルで testするか）— {{PROJECT_NAME}}

> `/tdd` が仕様の各受け入れ条件にレベルを割り当てるための指針。
> 原則: **下位レベルで testできるものは下位で**。E2E に寄せ過ぎない（遅く・脆い）。テストピラミッドを意識する。

## レベルの選び方

| レベル | 対象 | 例 | ツール |
|--------|------|----|--------|
| **unit** | 純粋ロジック・関数・1コンポーネント | 金額計算 / バリデーション / 整形 / 単一コンポーネントの表示・操作 | Vitest (+ RTL) |
| **integration** | 複数モジュール・route handler・DB/外部I/O 境界をまたぐ | API ルートが入力を検証し保存して応答する | Vitest（境界はモック or テスト用実体） |
| **E2E** | 画面をまたぐユーザー操作フロー | ログイン→一覧→詳細で目的を達成 | Playwright |

## 割り当ての指針

- 受け入れ条件が「**ある入力→ある出力**」で閉じる → **unit**。
- 「**複数の部品が連携して**正しく動く」→ **integration**。
- 「**ユーザーが画面を操作して**目的を達成できる」→ **E2E**（1機能あたり主要フロー1〜数本に絞る）。
- 同じ条件を複数レベルで重複検証しない。**一番安く検証できるレベル**を選ぶ。

## このプロジェクトでの適用（実際に用意した基盤を反映する）

> **「未実装・必要になったら導入」とは書かない。** `/tdd-init` で実行基盤を用意した状態を前提に、各レベルが今すぐ書ける前提で記述する。基盤の構成・起動手順は `test-infra.md`。

- unit: 常時利用（Vitest 導入済み）。
- integration（境界モック）: Vitest 上で外部境界（{{INTEGRATION_BOUNDARIES}}）をモック/差し替えして検証。追加基盤は不要。
- integration（実 DB）: {{INTEGRATION_DB_STATUS}}  <!-- 例: 「テスト DB(docker-compose.test.yml)＋globalSetup 用意済み」/「対象外（実 DB アクセスなし）」 -->
- E2E: {{E2E_STATUS}}  <!-- 例: 「Playwright＋storageState ログイン基盤 用意済み、e2e/ に配置」/「対象外（画面なし）」 -->

「対象外」と書くのは、本当にそのレベルが構造的に存在しない場合（画面が無い／DB が無い 等）だけ。**面倒・後回しを理由に「対象外」にしない。**
