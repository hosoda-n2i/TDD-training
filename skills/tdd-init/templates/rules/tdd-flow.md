# 仕様駆動 TDD の規律 — {{PROJECT_NAME}}

> `/tdd-init` が生成した常時適用ルール。このプロジェクトの開発はこの規律に従う。
> アプリの責務: {{DOMAIN_SUMMARY}}

## 基本姿勢

- **仕様 → テスト → 実装** の順。実装から書き始めない。
- 新機能・修正は **`/spec`（仕様を起こし精査）→ `/tdd`（実装）** の2段で進める。
- テストは仕様の受け入れ条件から導く。**「振る舞いを思いつく」のではなく「仕様から抽出する」**。
- テストを先に書き、**期待どおり赤**を確認してから実装する。
- **緑でない状態でコミットしない。** 実装はテストを通すためだけに書き、テストを実装に合わせて歪めない。
- **テスト無しで feature を実装しない。** 新機能・仕様変更は必ず `/spec`→`/tdd` を通す。「とりあえず実装」で TDD フローを飛ばさない。

## テストレベル（詳細は `test-strategy.md`）

- **unit（Vitest）**: 純粋ロジック・関数・コンポーネント単体。最も多く、速い。
- **integration**: 複数モジュール・route handler・DB 等をまたぐ。
- **E2E（Playwright）**: ユーザー操作フロー・画面横断の受け入れ条件。

仕様の各受け入れ条件は、検証に最も適したレベルに割り当てる。E2E で全部やろうとしない（遅く脆い）が、**逆に全部 unit に倒すのも禁止**。UI を伴うなら E2E、層またぎなら integration を必ず書く。実行基盤は `test-infra.md` に用意済み（無ければ用意してから書く）。

## 完了の定義（Definition of Done）

機能は、以下を満たすまで「完了」にしない:

- すべての受け入れ条件に、**割り当てたレベル**（unit/integration/E2E）のテストが存在し、緑である。
- UI/画面を伴う機能に **E2E が最低1本**ある（無いなら理由が仕様に明記されている）。
- API・DB・複数モジュールをまたぐ機能に **integration がある**（無いなら理由が明記されている）。
- 既存テストが回帰していない（全 unit 緑）。
- 型チェック / lint が通る。

unit だけで「完了」にしている場合、それは未完了。E2E/integration を省いた理由が言語化できないなら書く。

## フロー（`/tdd` が駆動する）

1. 仕様からテスト洗い出し（E2E シナリオを最初に地図化）
2. 実装計画立案（テストケース ↔ 実装箇所の対応）
3. テスト先行 → **テスト自体をレビュー**
4. 仕様内容の実装（最小実装 → リファクタ）
5. 既存テストの回帰確認
6. E2E / 各種テストの通過確認
7. 一通りの動作確認
8. 実行時不具合のレビュー → 修正 → 再実行

## 守ること

- コマンドは `.claude/tdd/commands.md` のものだけを使う。
- テストの配置・命名・モック方針は `.claude/tdd/rules/test-conventions.md` に従う。
- 確認ステップは最小限（実装計画の提示・E2E導入の可否）。それ以外は自己完結で進める。
- `push` はユーザーが明示したときだけ。コミットは受け入れ条件（論理単位）ごと。
